# Design: Nav UI Improvements and Structural Reads Edges

**Date:** 2026-06-25  
**Status:** Approved  
**Branch:** feat/ui-improvements  
**ADRs affected:** ADR-004b (amendment required ā€” structural reads)

---

## Overview

Three related improvements:

1. **Nav alphabetical sorting** ā€” sort at every level of the nav tree, including section groups inside aggregate nodes
2. **Nav group collapse** ā€” section groups (e.g. "Operations" under a DomainModel) get independent collapse/expand; the active node's groups auto-expand on navigation
3. **Structural reads edges** ā€” auto-generate `reads` edges at load time from `node-ref` properties pointing to external clusters; don't persist them in edge files; remove redundant `emitReadsEdge` calls from adapters; amend ADR-004b

Items 1 and 2 are pure frontend changes. Item 3 spans the loader, writer, adapters, and requires an ADR amendment.

---

## Item 1: Alphabetical Nav Sorting

### Problem

The nav tree sorts components alphabetically (`sortedComponents` in `NavTree`) and sorts template entries by `_sortKey`. Nodes within a template entry and nodes within navChildren groups are also sorted by ID. However, the **section group headers** themselves (e.g. "Operations", "Schemas" under a DomainModel entry) appear in Map insertion order ā€” whatever order the `/api/nodes` response happened to arrive in.

### Change

In `web/nav.js`, `buildNavTree` builds `navChildren` from `nestedByParent.get(node.id).values()`. Sort the resulting array by `label` before mapping:

```js
const navChildren = [...(nestedByParent.get(node.id)?.values() ?? [])]
  .sort((a, b) => a.label.localeCompare(b.label))
  .map(group => ({
    label: group.label,
    nodes: group.nodes.sort((a, b) => a.id.localeCompare(b.id)),
  }));
```

**Scope:** One line change in `web/nav.js`. No server changes.

## Item 2: Nav Group Collapse

### Problem

Section groups under aggregate node items (the "Operations", "Schemas" etc. groups rendered by `navChildren`) are always visible when their parent template entry is open. There is no per-group collapse toggle. The aggregate node row itself navigates on click ā€” a single click cannot serve as both expand and navigate.

### Interaction model

- Aggregate node row (`nav-node-item`) ā†’ click navigates (unchanged)
- Section group header (`nav-child-head`) ā†’ click collapses/expands that group only
- When navigating to a node: the new active node's groups auto-expand; all other nodes' groups collapse
- Manually collapsing a group on the active node is preserved until navigation changes the active node

### State model

Add `openGroupKeys` (`Set<string>`) to `NavTree`. Keys are `${parentNodeId}:${groupLabel}`.

**On `activeNodeId` change** (via `useEffect`): search the nav tree to find the node whose groups should open. That node is either:
- the top-level nav-node-item that *is* the active node (has `node.id === activeNodeId`), or
- the top-level nav-node-item that *contains* the active node as a navChild (`node.navChildren.some(g => g.nodes.some(c => c.id === activeNodeId))`)

Reset `openGroupKeys` to a Set of `${ownerNode.id}:${group.label}` for every group on that owner node. This collapses groups on all other nodes and expands the relevant owner's groups.

**On group header click**: toggle `${node.id}:${group.label}` in `openGroupKeys`.

This is consistent with the existing pattern: `openComponent` resets when a new component is selected; `openGroupKeys` resets when a new node is selected.

### Visual change

The `nav-child-head` element gains:
- A chevron icon (right when collapsed, down when expanded) 
- `cursor: pointer` style
- `onClick` handler

Content of the group is conditionally rendered based on whether its key is in `openGroupKeys`.

**Scope:** `web/app.jsx` only ā€” `NavTree` component. No server or nav.js changes.

## Item 3: Structural Reads Edges

### Problem

DomainModel cluster files declare fields with `$ref` values pointing to external cluster roots (e.g. `$ref: payruns.ValueObject.ApportionmentPolicy`). The cluster loader creates structural edges (has-field, has-value) from ownership hierarchy but does not create reads edges from `node-ref` property values. Without reads edges, `getClusterView` does not include external clusters in `includedNodes`, so `buildSchemaModel` cannot find referenced schemas, and field expansion (`canExpand = false`) fails in the detail view.

APIEndpoints imported from OpenAPI expand correctly because the OpenAPI adapter calls `emitReadsEdge` explicitly. Hand-crafted cluster files have no equivalent mechanism.

### Formalising the distinction (ADR-004b amendment)

ADR-004b defines `reads` as a semantic edge ā€” root-to-root, authored in edge files. This ADR amendment adds a third category of auto-generated edge: **structural reads**.

| Category | Types | Generated by | In edge files? |
|---|---|---|---|
| Structural ownership | `has-field`, `has-value` | cluster-loader (ownership hierarchy) | Never |
| **Structural reads** | `reads` | cluster-loader (node-ref properties) | Never |
| Semantic | `triggers`, `produces`, `reads`, `calls`, `implements`, `maps-to`, `derived-from`, `renamed-from` | Human-authored | Always |

`reads` therefore appears in both the structural and semantic categories: the loader generates structural reads that are never persisted; humans author semantic reads in edge files that are always persisted.

### Auto-generation rule

When the cluster loader materialises an owned child node, it inspects the child's template for properties with `format: node-ref`. For each such property whose value is a string not starting with `#/` (i.e. a global node ID, not a local ref):

1. Determine the **cluster root** ā€” the top-level root node of the current cluster being parsed
2. Emit a `reads` edge: `from = clusterRoot.id`, `to = globalNodeId`, marked `generated: true`
3. Deduplicate by edge ID: if a reads edge with the same `from__reads__to` already exists in the result, skip

The `from` is always the cluster root, keeping reads edges root-to-root per ADR-004b. Multiple fields referencing the same external cluster produce exactly one reads edge.

### Mark and filter

Add `generated?: true` to the `Edge` type in `src/schema/index.ts`. This field is internal ā€” never written to YAML.

`serializeGraph` in `src/writer/graph-writer.ts` extends its filter:

```ts
const explicitEdges = getAllEdges(graph)
  .filter(edge => !STRUCTURAL_EDGE_TYPES.has(edge.type) && edge.generated !== true)
```

This ensures auto-generated structural reads are never written to edge files, just as has-field and has-value are not.

### Adapter cleanup

`src/adapters/openapi/mapper.ts`: remove both `emitReadsEdge` calls (`emitSchemaNode` line ~348, `resolveFieldRef` line ~416). The cluster loader now generates these edges when processing the cluster YAML files that the import runner writes.

`src/adapters/asyncapi/mapper.ts`: same audit ā€” remove any `emitReadsEdge` equivalents.

### Import runner

No change required. The runner's existing `STRUCTURAL_EDGE_TYPES` filter (`has-field`, `has-value`) is for adapter output only; structural reads edges will never reach the runner because the adapters are no longer emitting them.

### Migration

Existing edge files may contain reads edges written by previous OpenAPI/AsyncAPI imports. These are harmless ā€” `getClusterView` follows both explicit and auto-generated reads edges, and `includedNodeIds` is a Set so no node is double-counted. Existing reads edges in edge files are not auto-cleaned (they are not `generated: true`). A future linter rule can flag and suggest removal of reads edges that are now structurally inferrable.

### Finding the cluster root during materialisation

`materialiseChildren` in `cluster-loader.ts` is currently called with `parent` ā€” which may be the root node or a nested child. To emit reads edges from the cluster root, pass the original root node down through the recursion:

```ts
function materialiseChildren(
  result: ClusterResult,
  parent: Node,
  clusterRoot: Node,   // NEW ā€” always the top-level root of this cluster
  source: Record<string, unknown>,
  templates: Map<string, Template>,
  filePath: string,
  diagnostics: Diagnostic[],
): void
```

Initial call: `materialiseChildren(result, root, root, record, templates, key, diagnostics)`.

### Template property inspection

The cluster loader already receives `templates: Map<string, Template>`. For each child node, call `getPropertySchemas(template.properties)` (same helper used in `server.ts`'s `annotateNodeRefProperties`) to find properties with `format: node-ref`. For each matching property with a non-local string value, emit the structural reads edge.

Helper to extract from server.ts into a shared location (`src/loader/template-utils.ts` or inline in cluster-loader):

```ts
function getNodeRefTargets(node: Node, template: Template): string[] {
  const propSchemas = getPropertySchemas(template.properties as Record<string, unknown>)
  const targets: string[] = []
  for (const [key, schema] of Object.entries(propSchemas)) {
    if ((schema as Record<string, unknown>).format !== 'node-ref') continue
    const value = node.properties[key]
    if (typeof value === 'string' && !value.startsWith('#/')) targets.push(value)
  }
  return targets
}
```
